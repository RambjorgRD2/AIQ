# Phase 01.1: Platform Migration - Research

**Researched:** 2026-03-29
**Domain:** Hosting migration (GitHub Pages → Vercel), Astro base path removal, custom domain DNS, Supabase auth redirect URLs
**Confidence:** HIGH

## Summary

This phase moves the AIQ site from GitHub Pages (at `https://RambjorgRD2.github.io/AIQ/`) to Vercel with the custom domain `www.theaiq.org`, and removes the `/AIQ` base path from every layer of the codebase. The migration is a pre-requisite for Phase 2 because Stripe webhooks require a serverless runtime — GitHub Pages is static-only and cannot receive webhooks.

The work falls into four cleanly-bounded areas: (1) Astro config changes — remove `base` and update `site`; (2) source code cleanup — replace all hardcoded `/AIQ/` strings and `BASE_URL` patterns; (3) Vercel project setup and GitHub Actions CI replacement; (4) Supabase auth redirect URL update in both the source code and the Supabase dashboard. DNS propagation for `www.theaiq.org` may take up to 48 hours after records are set, so DNS configuration should happen early in execution.

**Primary recommendation:** Use the official Vercel GitHub Actions workflow (pull/build/deploy with VERCEL_TOKEN) to replace the existing GitHub Pages deploy workflow. No Astro adapter is needed for the current static build. Remove `base` from `astro.config.mjs`, set `site` to `https://www.theaiq.org`, then do a targeted find-and-replace pass across the 20 affected source files.

## Architecture Patterns

### Pattern 1: Astro Config for Root Domain Deployment

When removing `base`, `import.meta.env.BASE_URL` becomes `'/'`. All `${BASE}/path` patterns in source code reduce to `/path`, which is correct for root deployment.

**Before (GitHub Pages):**
```javascript
// astro.config.mjs
export default defineConfig({
  site: 'https://RambjorgRD2.github.io/AIQ',
  base: '/AIQ',
  integrations: [tailwind()],
});
```

**After (Vercel root domain):**
```javascript
// astro.config.mjs
export default defineConfig({
  site: 'https://www.theaiq.org',
  integrations: [tailwind()],
});
```

No `base` key at all — do not set it to `'/'`, just omit it.

### Pattern 2: BASE variable becomes a no-op

Every page and layout currently has:
```javascript
const BASE = import.meta.env.BASE_URL.replace(/\/$/, '');
// At build time with no base set: BASE_URL = '/'  →  BASE = ''
```

With `base` removed, `BASE_URL` is `'/'`. After `.replace(/\/$/, '')`, `BASE` becomes `''` (empty string). This means `${BASE}/signin` becomes `/signin` — correct. The pattern continues to work correctly; no individual `${BASE}` interpolations need to change.

**Exception:** Hardcoded strings bypass this pattern entirely and must be replaced explicitly (see Runtime State Inventory).

### Pattern 3: Vercel GitHub Actions Deployment

Replace the existing `deploy.yml` (GitHub Pages) with Vercel's recommended workflow. Vercel requires three secrets:

- `VERCEL_TOKEN` — personal access token from vercel.com/account/tokens
- `VERCEL_ORG_ID` — from `.vercel/project.json` after `vercel link`
- `VERCEL_PROJECT_ID` — from `.vercel/project.json` after `vercel link`

The Supabase env vars (`PUBLIC_SUPABASE_URL`, `PUBLIC_SUPABASE_ANON_KEY`) already stored as GitHub Actions secrets must be injected into the Vercel build step as environment variables — OR configured in Vercel's dashboard under Project Settings → Environment Variables (so `vercel pull` fetches them automatically).

**Recommended workflow (`.github/workflows/deploy.yml` replacement):**
```yaml
name: Vercel Production Deployment
env:
  VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
  VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
on:
  push:
    branches:
      - main
  workflow_dispatch:
jobs:
  Deploy-Production:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - name: Install Vercel CLI
        run: npm install --global vercel@latest
      - name: Pull Vercel Environment Information
        run: vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}
      - name: Build Project Artifacts
        run: vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}
      - name: Deploy Project Artifacts to Vercel
        run: vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}
```

Note: `vercel pull` downloads environment variables from Vercel dashboard. If Supabase vars are added to Vercel's dashboard, they will be pulled here and baked into the build. Alternatively, inject them explicitly in the Build step:
```yaml
      - name: Build Project Artifacts
        run: vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}
        env:
          PUBLIC_SUPABASE_URL: ${{ secrets.PUBLIC_SUPABASE_URL }}
          PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.PUBLIC_SUPABASE_ANON_KEY }}
```

### Pattern 4: Vercel Project Setup (one-time)

Before CI can deploy, a Vercel project must exist and be linked:
1. Install Vercel CLI locally: `npm i -g vercel`
2. Run `vercel link` in project root → creates `.vercel/project.json`
3. Add `.vercel/project.json` to `.gitignore` (contains project IDs)
4. Copy `orgId` and `projectId` from `.vercel/project.json` to GitHub Secrets as `VERCEL_ORG_ID` / `VERCEL_PROJECT_ID`
5. Create `VERCEL_TOKEN` from vercel.com/account/tokens → add to GitHub Secrets

### Pattern 5: Vercel Custom Domain + DNS

For `www.theaiq.org` (subdomain):
- DNS record: `www` CNAME → `cname.vercel-dns-0.com`
- For apex `theaiq.org` (if needed): A record → `76.76.21.21`

Via Vercel CLI:
```bash
vercel domains add www.theaiq.org <project-name>
vercel domains inspect www.theaiq.org  # shows required records
```

Configure redirect: `theaiq.org` → `www.theaiq.org` in Vercel project settings (avoid duplicate content). SSL certificate is auto-provisioned after DNS verification (typically minutes).

### Pattern 6: Supabase Auth Redirect URL Migration

Two changes are required:

**A. Source code** — `src/pages/signin.astro` line 120:
```javascript
// Before
emailRedirectTo: 'https://RambjorgRD2.github.io/AIQ/auth/callback',

// After
emailRedirectTo: 'https://www.theaiq.org/auth/callback',
```

**B. Supabase dashboard** — Authentication → URL Configuration:
- Update **Site URL** from `https://RambjorgRD2.github.io/AIQ` to `https://www.theaiq.org`
- Add `https://www.theaiq.org/auth/callback` to the **Redirect URLs** allow list
- Optionally remove the old GitHub Pages callback URL after migration is confirmed working

The `emailRedirectTo` value in code must exactly match (or be a wildcard match for) an entry in the Supabase Redirect URLs allow list. Without this dashboard change, magic links will redirect to an unauthorized URL and auth will fail.

### Anti-Patterns to Avoid

- **Do not use `vercel.json` to rewrite URL paths** — Vercel docs explicitly state this produces inconsistent behavior with Astro. Use Astro's native redirect configuration instead.
- **Do not install `@astrojs/vercel` adapter for a static site** — it is only needed when using SSR or Vercel-specific features.
- **Do not set `base: '/'` in astro.config.mjs** — simply omit `base` entirely. Setting it to `'/'` is redundant and can cause confusion.
- **Do not remove GitHub Pages workflow before Vercel is confirmed working** — keep old workflow disabled (comment out trigger) until new domain is live and verified.

## Common Pitfalls

### Pitfall 1: Supabase Auth Fails After Domain Switch
**What goes wrong:** Users click magic links and get a Supabase error: "Redirect URL not allowed."
**Why it happens:** `emailRedirectTo` in `signin.astro` now points to `www.theaiq.org/auth/callback`, but the Supabase dashboard Redirect URLs allow list still only has the old GitHub Pages URL.
**How to avoid:** Update the Supabase dashboard (Authentication → URL Configuration) BEFORE or AT THE SAME TIME as the code change. Both changes must be live together.
**Warning signs:** Auth callback page shows an error immediately after deployment.

### Pitfall 2: GitHub Pages Still Serving After Switching to Vercel
**What goes wrong:** Old `https://RambjorgRD2.github.io/AIQ/` URL continues working (GitHub Pages remains active), causing confusion about which deployment is canonical.
**Why it happens:** GitHub Pages is configured at the repo level and doesn't auto-disable when Vercel is added.
**How to avoid:** After Vercel is confirmed working, disable GitHub Pages in the GitHub repo Settings → Pages → set Source to "None."
**Warning signs:** Pushing to `main` still triggers the old GitHub Pages workflow (check Actions tab).

### Pitfall 3: Hardcoded `/AIQ/` Strings Survive the Migration
**What goes wrong:** Sign-out redirects to `/AIQ/` (404), auth callback falls back to `/AIQ/` (404), share URL is wrong.
**Why it happens:** These 5 files have literal strings that don't use the `${BASE}` pattern.
**How to avoid:** Use the inventory table above — fix all 6 occurrences explicitly.
**Warning signs:** After deployment, signing out produces a 404; magic link callback fails to redirect.

### Pitfall 4: Vercel Build Misses Supabase Env Vars
**What goes wrong:** Build succeeds but `PUBLIC_SUPABASE_URL` / `PUBLIC_SUPABASE_ANON_KEY` are `undefined` at runtime; auth module throws.
**Why it happens:** `vercel pull` only fetches env vars from Vercel dashboard — if they were never added there, the vars are missing at build time.
**How to avoid:** Add `PUBLIC_SUPABASE_URL` and `PUBLIC_SUPABASE_ANON_KEY` to Vercel's dashboard (Project Settings → Environment Variables) OR inject them explicitly in the GitHub Actions workflow `env:` block.
**Warning signs:** Console errors about undefined Supabase URL/key on first load after migration.

### Pitfall 5: DNS Propagation Delays Block Testing
**What goes wrong:** `www.theaiq.org` returns NXDOMAIN or routes to old GitHub Pages for hours after DNS records are set.
**Why it happens:** DNS TTL and propagation delays — can be up to 48 hours for nameserver changes, but typically 5-30 minutes for CNAME changes to an existing domain.
**How to avoid:** Add DNS records early in the migration process. Test against the Vercel-provided `*.vercel.app` URL in the meantime — this URL is always available immediately and does not depend on DNS.
**Warning signs:** `dig www.theaiq.org` returns old values; Vercel dashboard shows domain as "Unverified."

### Pitfall 6: `.vercel/project.json` Committed to Git
**What goes wrong:** `orgId` and `projectId` are exposed publicly.
**Why it happens:** The `vercel link` command creates `.vercel/project.json` which is not in `.gitignore` by default.
**How to avoid:** Verify `.gitignore` includes `.vercel/` before committing after running `vercel link`.
**Warning signs:** Git status shows `.vercel/project.json` as a new untracked file.

## Open Questions

1. **Who controls the DNS for `theaiq.org`?**
   - What we know: Domain is `theaiq.org`, target subdomain is `www.theaiq.org`
   - What's unclear: Which registrar/DNS provider holds the zone (Cloudflare, GoDaddy, Namecheap, Vercel, etc.)
   - Recommendation: The planner must include a task that prompts the user to identify the DNS provider and add the CNAME record `www → cname.vercel-dns-0.com`. This is a human-action task, not automated.

2. **Should the apex domain `theaiq.org` redirect to `www.theaiq.org`?**
   - What we know: Standard practice is to pick one canonical domain and redirect the other. Vercel handles this natively.
   - What's unclear: User preference — `www` canonical vs. apex canonical
   - Recommendation: Default to `www` canonical (add A record `76.76.21.21` for apex with redirect to `www`). This is the Vercel-recommended configuration and allows CNAME usage for `www`.

3. **Should the old GitHub Pages URL (`RambjorgRD2.github.io/AIQ/`) remain live after migration?**
   - What we know: GitHub Pages continues serving until explicitly disabled in repo Settings
   - What's unclear: Whether the user wants a temporary redirect or immediate cutover
   - Recommendation: Keep GitHub Pages active until `www.theaiq.org` is confirmed working, then disable it. There is no easy way to redirect from GitHub Pages to a custom domain — it will simply become a dead URL. Since no external links to the old URL are documented, this is acceptable.

4. **Where should `PUBLIC_SUPABASE_URL` and `PUBLIC_SUPABASE_ANON_KEY` live in the new CI setup?**
   - What we know: Currently stored as GitHub Actions secrets, injected in the Build step.
   - Recommendation: Add them to Vercel's dashboard (Project Settings → Environment Variables → Production) so `vercel pull` fetches them. This is the canonical Vercel approach. Also keep them in GitHub Secrets for the GitHub Actions workflow's `env:` block as a belt-and-suspenders fallback.
